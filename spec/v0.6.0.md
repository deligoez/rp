# Spec: Replace `deps` with `install` and `update` commands

**Version:** v0.6.0
**Type:** Breaking change â€” command and manifest restructuring

## Motivation

The current `deps` command runs a single list of shell commands per repo, used for initial dependency installation. But repos have two distinct lifecycle phases:

- **install** â€” run once after cloning (e.g. `composer install`, `npm install`)
- **update** â€” run periodically to keep dependencies current (e.g. `composer update`, `npm update`)

Splitting `deps` into `install` and `update` makes the intent explicit and enables `rp up` to run the right commands depending on whether a repo was just cloned or already existed.

## Scope

1. Replace `deps` manifest key with `install` and `update`
2. Replace `rp deps` command with `rp install` and `rp update`
3. Update `rp up` to run `install` on newly cloned repos and `update` on existing repos
4. Replace `--no-deps` flag with `--no-install` and `--no-update` on `rp up`
5. Update JSON output: `deps` key â†’ `install` and `update` keys in `rp up`
6. Update all tests
7. Update documentation (README.md, CLAUDE.md)

## Design

### 1. Manifest format changes

#### 1.1 YAML key rename

Replace `deps:` with two optional keys: `install:` and `update:`.

**Before (v0.5.0):**
```yaml
- repo: deligoez/blog
  deps:
    - composer install
    - npm install
```

**After (v0.6.0):**
```yaml
- repo: deligoez/blog
  install:
    - composer install
    - npm install
  update:
    - composer update
    - npm update
```

Both keys are optional. A repo can have `install` only, `update` only, both, or neither.

#### 1.2 Manifest parsing (`internal/manifest/manifest.go`)

Update `rawRepo` struct:
```go
type rawRepo struct {
    Repo    string   `yaml:"repo"`
    Install []string `yaml:"install"`
    Update  []string `yaml:"update"`
}
```

Update `RepoEntry` struct:
```go
type RepoEntry struct {
    Repo      string
    Owner     string
    Category  string
    LocalPath string
    CloneURL  string
    Install   []string // shell commands for initial setup
    Update    []string // shell commands for dependency updates
}
```

Remove the `Deps` field entirely. Update `parseOwnerNode()` to populate `Install` and `Update` instead of `Deps`.

#### 1.3 Validation rules

Update R7:
- **Before:** `deps` entries must be non-empty strings
- **After:** `install` and `update` entries must be non-empty strings

Same validation logic, applied to both fields independently.

**`deps` key rejection:** If a repo entry contains a `deps:` key, reject with `HintError`. This detection must happen at the `yaml.Node` level (scanning mapping keys in `parseOwnerNode`) because `rawRepo` no longer has a `Deps` field, so `yaml.Unmarshal` would silently ignore the unknown key.
- **Error:** `repo "<name>" uses removed key "deps"`
- **Hint:** `Rename "deps:" to "install:" (and optionally add "update:").`

Validation is fail-fast: the first repo with `deps:` stops parsing.

### 2. Command changes

#### 2.1 Remove `rp deps` command

Delete `cmd/deps.go` entirely. Add a hidden `deps` cobra command that prints a migration hint and exits with code 2:
```
"rp deps" has been replaced by "rp install" and "rp update".
```
This ensures users upgrading from v0.5.0 get a clear message instead of cobra's generic "unknown command" error.

#### 2.2 Add `rp install` command (`cmd/install.go`)

New command that runs `install:` commands for each repo.

**Behavior:**
- Iterates all repos (or filtered via `--filter` / positional arg)
- Skips repos without `install:` defined
- Skips repos not on disk (warning on stderr)
- Runs each `install:` command via `sh -c` in the repo's directory (fail-fast: stops on first failed command per repo)
- Supports `--dry-run`, `--json`, `--compact`, `--filter`
- Positional arg `rp install [repo]` overrides `--filter`
- Exit codes: 0 = all succeeded, 1 = at least one skipped (not on disk), 2 = at least one command failed. Precedence: highest value wins.

**Human output:**
```
owner
  category/repo     OK command1
  category/repo     OK command2

-- Summary --
N repos, M commands succeeded, K skipped, J failed
```

**JSON output:**
```json
{
  "command": "install",
  "exit_code": 0,
  "summary": {"succeeded": 1, "skipped": 0, "failed": 0, "total": 1, "commands": 2},
  "repos": [
    {
      "repo": "deligoez/blog",
      "status": "ok",
      "commands": [
        {"command": "composer install", "status": "ok"},
        {"command": "npm install", "status": "ok"}
      ]
    }
  ]
}
```

#### 2.3 Add `rp update` command (`cmd/update.go`)

Identical structure to `rp install`, but uses `update:` commands instead.

**Behavior:**
- Same as `rp install` except reads `Update` field
- Skips repos without `update:` defined (no error, just skip)
- Same flags: `--dry-run`, `--json`, `--compact`, `--filter`
- Positional arg `rp update [repo]` overrides `--filter`
- Same exit codes and precedence rules

**JSON output:**
```json
{
  "command": "update",
  "exit_code": 0,
  "summary": {"succeeded": 1, "skipped": 0, "failed": 0, "total": 1, "commands": 2},
  "repos": [...]
}
```

#### 2.4 Internal command runner

Since `rp install` and `rp update` have identical execution logic (only the command list source differs), extract a shared runner.

Rename `internal/deps/deps.go` â†’ `internal/runner/runner.go`:
```go
package runner

func RunCommands(path string, commands []string) error
```

Same implementation as current `RunDeps` (fail-fast: stops on first error, returns that error).

**Per-command JSON granularity:** Callers (`cmd/install.go`, `cmd/update.go`, `cmd/up.go`) invoke `RunCommands` **one command at a time** (single-element slice) so they can build per-command status entries for JSON output. This matches the current pattern in `cmd/up.go` where `deps.RunDeps` is called per-command.

Delete `internal/deps/` package entirely.



### 3. `rp up` changes (`cmd/up.go`)

#### 3.1 Phase restructuring

Current phases: bootstrap â†’ sync â†’ deps
New phases: bootstrap â†’ sync â†’ install (new clones) â†’ update (existing repos)

**Data flow:** The bootstrap phase returns both a **cloned set** and a **failed set** of repo names. This three-way partition determines install/update targets:
- **Newly cloned** (bootstrap succeeded) â†’ eligible for install phase
- **Failed to clone** (bootstrap failed) â†’ excluded from both install and update
- **Pre-existing** (not in cloned or failed sets) â†’ eligible for update phase

**Phase 3 â€” Install:** Run `install:` commands only on repos that were **successfully cloned in the bootstrap phase** of this `rp up` invocation. Skip repos without `install:` defined. Newly cloned repos are excluded from the update phase.

**Phase 4 â€” Update:** Run `update:` commands on **pre-existing repos** (existed before bootstrap), regardless of sync outcome (synced, already up-to-date, skipped-dirty, skipped-unpushed). Skip repos without `update:` defined. Repos that failed to clone are excluded.

#### 3.2 Flags

- Remove `--no-deps`
- Add `--no-install` â€” skip install phase
- Add `--no-update` â€” skip update phase
- Both flags can be combined (skips both phases)

When a phase is skipped:
- Human output: the phase section header (`== Install ==` / `== Update ==`) is omitted entirely
- JSON output: the key is present but set to `null`

#### 3.3 Human output

```
== Bootstrap ==
...
== Sync ==
...
== Install ==
owner
  category/repo     OK composer install
...
== Update ==
owner
  category/repo     OK composer update
...
```

#### 3.4 JSON output

Update `UpResult` in `internal/output/output.go`: replace `Deps *SubResult` with:
```go
Install *SubResult `json:"install"`
Update  *SubResult `json:"update"`
```

**No `omitempty`** on these fields â€” when a phase is skipped via flag (`--no-install`/`--no-update`), the value is `null`. When a phase runs but finds zero eligible repos, it produces a `SubResult` with an empty repos array and zeroed summary counts (not `null`).

```json
{
  "command": "up",
  "exit_code": 0,
  "bootstrap": {...},
  "sync": {...},
  "install": {
    "summary": {"succeeded": 1, "skipped": 0, "failed": 0, "total": 1, "commands": 2},
    "repos": [...]
  },
  "update": {
    "summary": {"succeeded": 3, "skipped": 1, "failed": 0, "total": 4, "commands": 5},
    "repos": [...]
  }
}
```

#### 3.5 Exit code composition

The composite exit code for `rp up` is the **maximum** across all phases: `max(bootstrap_exit, sync_exit, install_exit, update_exit)`. This extends the existing `upExitCode` function to include the two new phases.

#### 3.6 Dry-run

`--dry-run` applies to all phases. In dry-run mode:
- Bootstrap phase reports `would_clone` status (no actual cloning)
- Install phase targets repos with bootstrap status `would_clone` (treats them as "newly cloned" for partitioning purposes)
- Update phase targets repos that **already exist** on disk (pre-existing)
- JSON status for dry-run commands: `"would_run"`
- Human output shows `would run <command>` per repo
