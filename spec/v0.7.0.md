# Spec: Add `rp validate` command

**Version:** v0.7.0
**Type:** Additive feature — new command

## Motivation

Users frequently edit the manifest file by hand, especially after breaking changes like v0.5.0 (`owners` wrapper removal) and v0.6.0 (`deps` → `install`/`update`). Today the only way to check if a manifest is valid is to run a command with side effects (`rp list`, `rp status`), which succeeds or fails based on manifest structure.

A dedicated `rp validate` command gives users a safe way to check manifest correctness without side effects, and produces a clean summary of what was parsed.

## Scope

1. Add new `rp validate [path]` command
2. Optional positional arg for alternative manifest path
3. Human output: `OK 87 repos valid`
4. JSON output with structured summary
5. Tests + QA regression tests
6. Documentation updates (README.md, CLAUDE.md)

## Design

### 1. Command behavior

`rp validate` loads the manifest and reports whether it is structurally valid.
**Path resolution precedence:**
1. Positional arg (if provided)
2. `-m` / `--manifest` flag
3. `RP_MANIFEST` env var
4. Default: `~/.config/rp/manifest.yaml`

The positional arg receives the same path expansion as `-m` (tilde expansion, absolute/relative resolution). When both positional arg and `-m` are provided, the positional arg wins (with no warning — intentional).

Example:
```
rp validate                           # validates default manifest
rp validate /path/to/other.yaml       # validates /path/to/other.yaml
rp validate --json                    # structured output
```

**`--filter` is ignored:** validation is a whole-manifest operation. Passing `--filter` produces no warning and no effect, consistent with how `manifest init` treats it.

### 2. Human output

**On success** (stdout):
```
OK 87 repos valid
```

Exact format: `{OK_SYMBOL} {n} {repo|repos} valid` where:
- `{OK_SYMBOL}` is `ui.SymbolOK()` (colored "OK" when TTY, plain "OK" otherwise)
- `{n}` is the repo count
- `{repo|repos}` uses `ui.Plural()` for correct singular/plural

Success output is written to **stdout**. In `--json` mode, no human output is emitted (JSON goes to stdout).

**On failure** (stderr): Use the existing `output.FormatHumanError()` pattern to render `HintError`:
```
error: <error message>
hint:  <hint text>
```

**Missing manifest hints:**
- Default path missing → hint: `"create manifest with: rp manifest init --dir ~/Developer"` (same as existing `Load()` hint)
- Explicit path missing (positional arg) → same hint (Load already handles this uniformly)

### 3. JSON output

**Success:**
```json
{
  "command": "validate",
  "exit_code": 0,
  "summary": {
    "valid": true,
    "repos": 87,
    "owners": 5,
    "categories": 8,
    "install_commands": 23,
    "update_commands": 23
  }
}
```

**Summary field meanings:**
- `valid`: always `true` on exit 0 (sentinel for agents to key off without parsing exit_code)
- `repos`: count of all parsed repo entries (regardless of clone state — validate does not touch disk)
- `owners`: count of top-level owner groups (`len(Manifest.Owners())`)
- `categories`: count of category keys across all categorized owners (flat owners contribute 0; computed as sum of `len(ownerGroup.Categories)` for categorized owners)
- `install_commands`: total count of install command lines across all repos (sum of `len(r.Install)`). Example: if repo A has 2 install lines and repo B has 3, the value is 5.
- `update_commands`: total count of update command lines across all repos (sum of `len(r.Update)`). Same counting rule as above.

No `repos` array in the top-level result (unlike `status`/`list`). The `--compact` flag is a no-op for validate since there is no array to omit.

The global `--filter` flag is inherited by cobra but ignored at runtime — validation is always whole-manifest.


### 4. Exit codes

- `0`: manifest is valid
- `2`: any parse/validation error

`validate` intentionally skips exit code `1` because structural validity is binary — there is no partial-success state. Other commands use exit 1 for "attention needed" (dirty/missing repos), which doesn't apply to a pure parse check.


### 5. Command registration (`cmd/validate.go`)

New file `cmd/validate.go`:

```go
var validateCmd = &cobra.Command{
    Use:   "validate [path]",
    Short: "Validate the manifest file structure",
    Args:  cobra.MaximumNArgs(1),
    RunE:  runValidate,
}
```

`runValidate` flow:
1. Determine path: positional arg if provided, else `ManifestPath` (which already encodes `-m` flag > `RP_MANIFEST` env > default via `root.go`)
2. Call `manifest.Load(path)` — this runs all existing validation
3. On error:
   - JSON mode: `output.PrintErrorAndExit("validate", err)` — emits ErrorResult with exit code 2
   - Human mode: write `output.FormatHumanError(err)` to stderr, then `os.Exit(2)`
4. On success: compute counts, emit JSON via `output.PrintAndExit(SuccessResult{...})` or print human line to stdout, exit 0

### 6. Tests

#### 6.1 Integration tests (`cmd/json_test.go`)

Add the following tests:

| Test | What it verifies |
|------|------------------|
| TestValidateValidManifest | Valid manifest → exit 0, JSON has `valid: true` and correct counts |
| TestValidateYAMLParseError | Manifest with malformed YAML → exit 2, JSON has parse error + hint |
| TestValidateSemanticError | Manifest with structural error (e.g. empty category) → exit 2, JSON has validation error + hint |
| TestValidatePositionalPath | `rp validate <path>` uses the given path, not default |
| TestValidatePositionalOverridesFlag | `rp validate <path> -m <other>` uses positional path (precedence) |
| TestValidateDepsKeyRejected | Manifest with `deps:` key → exit 2, migration hint |
| TestValidateMissingManifest | Non-existent path → exit 2, "create manifest" hint |
| TestValidateHumanOutput | Non-JSON mode, NO_COLOR=1 → stdout contains exact `OK N repos valid` (with correct plural) |
| TestValidateSingularRepo | Manifest with 1 repo → human output uses singular `1 repo valid` |
